## Commands

```bash
npm run dev        # Next.js dev server (Turbopack)
npm run build       # Production build — the only check that catches async/Server Action errors (see below)
npm run test:run    # Vitest, single run
npm run lint        # ESLint
```

## Architecture

- Route groups: `src/app/(auth)/` (login/register, public) and `src/app/(app)/` (dashboard/pelanggan/pengaturan/pesanan, behind auth + role checks). Layout for `(app)` composes `AppShell` (client) wrapping `Sidebar`/`TopBar`/children.
- Supabase is **remote-only** — there is no local CLI/`supabase start` workflow. Schema changes go straight to the live project (ref `pjkddahrjjqblexxhaef`, region ap-southeast-1) via the Supabase MCP tools (`apply_migration`, `execute_sql`, `get_advisors`); migration files in `supabase/migrations/` are a record of what's already been applied, not a local source of truth to replay.
- `src/lib/supabase/server.ts` / `client.ts` / `admin.ts` are the only places Supabase clients should be constructed (server component / browser / service-role respectively); `require-owner.ts` wraps the owner-role check used by gated Server Actions.
- `settings` table is a simple key-value store (columns: `key`, `value` text). Currently one seed row: `pesanan_locked = 'false'`. Readable by any authenticated user; writable by owner only (RLS). Read it via the server client, not admin.
- **Realtime publication contains `pesanan` only.** `item_pesanan` and `pembayaran` must NEVER be added — their rows contain price/payment data that would leak to helpers via the websocket before the role-aware refetch. `RealtimeRefresh` uses the Realtime event only as a "something changed" signal and refetches via a server query.
- **Pesanan status model**: three values only — `diproses`, `selesai`, `dibatalkan`. New pesanan always inserts as `diproses`. Only the owner can transition status. Helpers have no status-change actions.

## Environment

`.env.example` lists the 3 required vars: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` (service-role key is server-only, used by `admin.ts` for user-management actions like `createUser`/`deleteHelper`).

## Codebase conventions

- **`'use server'` files: every export must be `async`.** A sync helper in a Server Actions file compiles fine under `tsc`/`eslint`/`vitest` but breaks `npm run build` ("Server Actions must be async functions"). Always run `npm run build` before calling Server Action changes done — it's the only check that catches this.
- **shadcn here uses Base UI (`@base-ui/react`), not Radix** (`components.json` → `"style": "base-nova"`). Composed components take a `render={<Button .../>}` prop instead of `asChild`. Don't hand-write Radix-pattern code from memory — inspect the generated component after `npx shadcn@latest add <name>`.
- **Owner-gated Server Actions**: use `requireOwner(supabase)` from `src/lib/supabase/require-owner.ts` rather than hand-rolling the auth+role check inline. RLS is the real authorization boundary; this is defense-in-depth.
- **`<form action={fn}>` requires `fn` to return `void | Promise<void>`.** Actions here return `{ error?: string }` for error surfacing, so form-triggered deletes/mutations need a small client component with an `onClick` handler instead (see `DeletePaymentButton.tsx`).
- **Popup-blocker-safe pattern for print/PDF flows**: call `window.open('', '_blank')` synchronously as the *first* statement in the click handler, before any `await` — navigate the captured window reference once async work finishes. Calling `window.open` after an `await` gets blocked by browsers.
- **Never pass a `next/dynamic`-wrapped component into `@react-pdf/renderer`'s `pdf()`.** `next/dynamic` wraps a component in `React.lazy` + Suspense, which only `react-dom` knows how to resolve. `pdf()` uses its own non-DOM reconciler and doesn't support Suspense, so it silently renders broken/empty output. If you need code-splitting for a PDF component, resolve it with a plain `await import('...')` inside the click handler instead, then pass the real component to `pdf()`.
- **RLS helper functions that query a table protected by a policy calling that same function need `SECURITY DEFINER`.** Without it the function's internal `SELECT` re-triggers the calling policy → infinite recursion. It may look fine for authenticated users (their row short-circuits the `OR`) but blows up for anonymous requests. Verify with `curl /rest/v1/<table>` using the anon key — `npm run build`/tests won't catch this.
- **Sequence-generating `SECURITY DEFINER` functions (e.g. `kode_pesanan` numbering) should use `INSERT ... ON CONFLICT DO UPDATE ... RETURNING ... INTO`, not a separate `SELECT`/`UPDATE ... WHERE` pair.** A two-statement read-then-write can return zero rows (no existing row for this period yet) and silently produce `NULL`, which then violates a downstream `NOT NULL` constraint. An upsert always affects exactly one row, structurally eliminating that failure class.
- **Postgres check constraints are enforced per-statement, not deferred, even within one migration transaction.** If a migration both backfills a column to satisfy a new shape and drops/relaxes an old constraint that the backfill would transiently violate, the constraint drop must run *before* the backfill `UPDATE`, regardless of the fact that the constrained column or table is dropped later in the same migration.
- **When a migration's backfill `UPDATE` touches a table guarded by a write trigger that assumes a real session (`auth.uid()`), wrap the backfill in `alter table ... disable trigger <name>` / `... enable trigger <name>`.** Migrations run without a real authenticated user, so role-checking triggers will reject the backfill otherwise.
- **Shared client-side UI state needed by sibling Server Components in a layout (e.g. a mobile sidebar-drawer toggle shared between the sidebar and the top bar) can't live in the Server Component layout itself.** Lift it into a small client wrapper component (e.g. `AppShell.tsx`) that holds the `useState` and composes the Server-rendered children as props/`children`, rather than trying to thread state through the Server Component tree.
- **Every UI change must be checked on mobile too, not just desktop.** This app has no desktop-only audience — tables need a `sm:hidden` card layout alongside the `hidden sm:block` table (see `OrderList.tsx`), and any inline editor (e.g. `ItemsSection.tsx`) needs to be checked for how it behaves squeezed into a narrow column/card, not just a wide desktop layout. Don't consider a UI task done until you've reasoned through (or viewed) it at a mobile viewport width.
- **`pg_trigger_depth()` bypass is required when a cascade trigger calls a write-guard trigger on a parent table.** For example, `bump_pesanan_updated_at_from_item` (SECURITY DEFINER, fires on `item_pesanan` writes) does `UPDATE pesanan SET updated_at = now()`, which re-fires `guard_pesanan_write`. That guard runs with the original request's `auth.uid()`, so a helper updating their own item would get rejected because they don't own the pesanan. Fix: add `IF pg_trigger_depth() > 1 THEN RETURN NEW; END IF;` at the top of any write guard that may be invoked by a cascade. This is a latent bug to watch for whenever a trigger on table B modifies table A that has its own write guard.
- **Owner bypasses DB write-guard triggers — all mutating Server Actions must enforce status checks at the app layer too.** `guard_item_pesanan_write` blocks writes on selesai/dibatalkan pesanan for helpers, but owners bypass it entirely. Without an explicit `status === 'diproses'` check in the action, an owner can mutate items on a closed pesanan. Apply this check in every action that writes `item_pesanan`, including the checklist toggles and bulk price update.
- **Client-supplied foreign-key IDs in Server Actions are untrusted for authorization.** When an action is keyed by an item ID but needs the parent `pesanan_id` (e.g. for revalidatePath or a status check), look up the real `pesanan_id` from the DB using the item ID — don't trust the `pesananId` the client passes alongside. A malicious or stale client value could bypass the status check or leak a path it shouldn't know about.
- **Fail-closed pattern for settings-based gates**: when a Server Action reads a `settings` row to decide whether to allow an operation (e.g. `pesanan_locked`), treat any DB error or missing row as a blocking condition — return an error rather than silently proceeding. A missing row means the table wasn't seeded correctly, which is an infrastructure problem that should surface, not be swallowed as "not locked."
- **`pesanan_locked` scope — helpers only, not owners.** The setting locks all helper mutations (add/edit/delete items, toggle checklists, reset checklist). Owners are only locked by `statusLocked` (order status). In `[id]/page.tsx` keep `statusLocked` and `isLocked` as separate variables; use `statusLocked` for owner-only UI like `BulkPriceForm` and the owner Reset button, and `isLocked` (status OR settings-lock) for helper-facing UI. The `checkHelperLock()` helper in `actions.ts` centralizes the server-side check.
- **`type="number"` inputs must store raw strings in state, not numbers.** `Number("") === 0` causes the input to re-render as `"0"`; on mobile the next keystroke appends instead of replaces, producing `"015"`. Pattern: `onChange={(e) => setState(e.target.value)}`, parse with `parseInt`/`parseFloat` only at save/calculation time, and initialise with `''` (not `0` or `1`) so the field starts empty. See `BulkPriceForm.tsx` and `ItemsSection.tsx`.
- **Postgres RLS restricts rows, not columns — it can't hide specific fields (e.g. prices) from one app role while allowing another role to read the rest of the same row.** All authenticated users share one Postgres role (`authenticated`); the `role` column distinguishing owner/helper is just data. To keep price data out of a helper's hands, don't fetch `harga_satuan`/`diskon`/`subtotal`/`pembayaran` in Server Components for non-owners at all — an unfetched field never reaches the RSC payload sent to the browser. This is an app-layer guarantee only — RLS gates rows, not columns, so a helper hitting the REST API directly with the anon key can still read price columns. Full column masking would require a `SECURITY DEFINER` view with `INSTEAD OF` triggers; not implemented.
- **Toggle buttons must show the action, not the current state.** A button labelled with what IS (e.g. "Terkunci") confuses users — they can't tell if clicking will change or confirm that state. Label with what WILL HAPPEN: "Kunci" when currently unlocked, "Buka Kunci" when currently locked. This is a recurring mistake; apply it to any toggle-style button in this codebase.
- **Multiple URL-param inputs must not each call `router.push` on `onChange`.** Each push triggers a navigation; the next push reads `useSearchParams()` before the previous navigation commits, so the second field's push overwrites the first field's value with the stale empty string. Pattern: hold both values in local `useState`, update on `onChange`, and push to the router on `onBlur`. See `DashboardDateFilter.tsx`.
- **Fixed/floating elements (FAB) need matching bottom padding on the scroll container.** The "Pesanan Baru" FAB is `fixed bottom-6 right-6` and covers page content on mobile. The `<main>` in `AppShell.tsx` uses `pb-24 sm:pb-6` to create clearance. Add the same offset whenever a new fixed element is introduced.
- **`sisaTagihan === 0` does not mean "Lunas" when `totalPesanan` is also 0.** An order whose items have no price yet has `sisaTagihan = 0 - 0 = 0`, which would show "✓ Lunas" — a false signal. Always gate the "Lunas" display on `totalPesanan > 0 || pembayaranList.length > 0`; otherwise show "Belum ada harga". See the Pembayaran section in `[id]/page.tsx`.
- **Client-component pages cannot export `metadata`.** `'use client'` files silently ignore `export const metadata`. Wrap the page in a server-component `layout.tsx` for the route group (e.g. `src/app/(auth)/layout.tsx`) and export `metadata` there instead.
- **`@react-pdf/renderer` images in browser context require base64 data URIs, not relative paths.** `pdf()` runs in its own non-DOM reconciler and cannot reliably resolve `/public/image.jpg` paths. Fetch images before calling `pdf()` and convert them: use `canvas.toDataURL('image/png')` for images that need cropping/transformation, or `FileReader.readAsDataURL(blob)` for images served as-is. A4 page dimensions for absolute positioning: 595.28 × 841.89 PDF points; use `position: 'absolute'` + `opacity` on an `<Image>` for watermarks.
- **`autoFocus` pattern for dynamically added list items.** To focus the first input of a newly added item (e.g. "Tambah Barang" adding a row), track the last-added item's id: `const [lastAddedId, setLastAddedId] = useState<string | null>(null)`, set it when creating the item, then pass `autoFocus={item.id === lastAddedId}` to the input. Since `autoFocus` fires only on initial mount, it focuses exactly the new item without disturbing others.
- **Deduplicate repeated auth/role DB queries with React `cache()`.** The `(app)` layout and every page both need the current user's role. Without caching, each Server Component fires its own round-trip. Wrap the fetch in `cache()` from `'react'` (`src/lib/supabase/request-cache.ts` exports `getCurrentUser`, `getAuthUser`, `getPesananLocked`); React deduplicates calls with the same arguments within one render tree, so only one DB query fires per request regardless of how many Server Components call them. **Do not call these from Server Actions** — Actions run outside the render tree and don't share the cache scope.
- **Parallelize independent Supabase queries in Server Components with `Promise.all()`.** Any two queries whose results don't depend on each other should run concurrently. Pattern: `const [{ data: a }, { data: b }] = await Promise.all([supabase.from('a').select(...), supabase.from('b').select(...)])`. Common cases: fetching a pesanan + the lock setting in `[id]/page.tsx`; fetching the user list + lock setting in `pengaturan/page.tsx`; the two pesanan queries (all-time vs period) in `dashboard/page.tsx`.
- **Replace two-query item→pesanan lookups with a single join.** `getItemPesananStatus` previously did two sequential queries (fetch item to get `pesanan_id`, then fetch pesanan to get `status`). Use a nested select instead: `supabase.from('item_pesanan').select('pesanan_id, pesanan:pesanan(status)').eq('id', itemId)` — one round-trip, same data. Apply this pattern whenever you need parent-table data keyed by a child-table row.
- **Replace sequential N+1 loops with `Promise.all()` in Server Actions.** Loops like `for (const item of items) { await supabase.update(...) }` do N sequential round-trips. Replace with `await Promise.all(items.map(item => supabase.update(...)))` to run all updates concurrently. Used in `updateAllItemHarga` and `clearAllPelanggan`. If any single update failing should abort the whole operation, check `results.find(r => r.error)` after the `Promise.all`.
